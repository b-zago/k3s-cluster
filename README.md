## Repo structure

```
├── charts                     # one Helm chart per app (generated by rika)
│   ├── rikami-api
│   │   └── templates
│   ├── portfolio
│   │   └── templates
│   └── ...
└── cluster
    ├── root-app.yaml          # app-of-apps root, watches cluster/
    ├── infra.yaml             # watches cluster/infra/
    ├── workloads-root.yaml    # watches cluster/workloads/
    ├── infra                  # essential cluster services
    │   ├── cluster-config     # gateways, cert issuers, argocd config, roles
    │   └── monitoring         # prometheus / grafana stack
    └── workloads              # appset, namespaces, postgres clusters, secrets
```

## Tech overview

- **ArgoCD** as the GitOps engine reconciling the whole repo
- **Traefik** as Gateway API and Ingress controller
- **Cert Manager** for automatic TLS via Let's Encrypt DNS-01 (Cloudflare)
- **Sealed Secrets** to safely keep secrets in the repo
- **kube-prometheus-stack** (Prometheus, Grafana, Alertmanager) as the monitoring stack
- **CloudNativePG** to run PostgreSQL clusters
- **Atlas Operator** for declarative database schema operations

## App of apps pattern

ArgoCD watches and applies every manifest in the repo on its own via the app-of-apps pattern.

It all starts with [root-app.yaml](./cluster/root-app.yaml), which wraps the whole [cluster/](./cluster/) directory. Every committed change inside `cluster/` gets picked up and synced, and since the root manifest lives in that directory too, it also watches itself.

Below the root sit two branches:

- [infra.yaml](./cluster/infra.yaml) watches [cluster/infra/](./cluster/infra/), the essential services the cluster needs to function (Traefik, Cert Manager, Sealed Secrets, CloudNativePG, Atlas Operator), plus nested apps for [cluster-config](./cluster/infra/cluster-config/) and [monitoring](./cluster/infra/monitoring/). Upgrading something like Traefik is just a version bump in its manifest; ArgoCD detects the change and rolls it out. Adding a new essential resource is just a new manifest in the directory.
- [workloads-root.yaml](./cluster/workloads-root.yaml) watches [cluster/workloads/](./cluster/workloads/), which holds the application namespaces, Postgres clusters and the ApplicationSet that deploys my actual apps.

I really like this approach since it lets me add/modify/remove manifests locally and push when I'm ready, leaving the rest to ArgoCD. **But most importantly** the repo is the exact description of what is currently applied on the cluster, so when something unexpected happens I can always revert.

## CI/CD

Pretty straightforward - GitHub Actions build the docker image with a sha tag, then the tag gets updated inside the matching `values-*.yaml` in [charts/](./charts/). ArgoCD sees the change and applies it.

I use reusable workflows and actions across my repos. You can find them [here](https://github.com/b-zago/actions).

## Deployments

_This is where the fun begins_

Every app in [charts/](./charts/) is a thin Helm chart that depends on my [rikami-charts](https://github.com/b-zago/rikami-charts) library and ships one `values-<env>.yaml` per environment. The charts aren't hand-written - they're generated by [rika](https://github.com/b-zago/rikami), my own CLI that turns a small declarative `rikami.yaml` manifest into a ready-to-deploy chart, handling templating, multi-environment values and secret sealing along the way.

The flow:

1. Describe the app in `rikami.yaml` using reusable building blocks (shards, fragments, confs) from [rikami-base](https://github.com/b-zago/rikami-base).
2. `rika manifest` fetches those templates, renders the chart, and seals any secrets automatically.
3. The generated chart lands in [charts/](./charts/) and gets committed to this repo.
4. ArgoCD's appset picks up the new `values-*.yaml` and rolls it out.

The [rikami-api](https://github.com/b-zago/rikami-api) serves the sealed-secrets cert and is growing toward fully automated, API-driven deploys. For everything `rika` can do, see the [rikami README](https://github.com/b-zago/rikami).

### Why?

Pretty much every app I deploy is the same shape:

**HTTPRoute** -> **Service** -> **Deployment**

I got tired of copying near-identical manifests and tweaking them slightly over and over, so rikami captures those patterns once and stamps them out from a few lines of config.

### ArgoCD syncs

[appset.yaml](./cluster/workloads/appset.yaml) watches `charts/*/values-*.yaml`. When a new file shows up it creates an application and deploys it to the matching namespace (`staging` / `prod`).

## Secrets management

I manage all my secrets with **Sealed Secrets** [(project repo)](https://github.com/bitnami-labs/sealed-secrets) so they can live safely in the repo. It encrypts secret values with a public key that only the controller in my cluster can decrypt with its private key.

### Implemented

- Automated secret encryption through my own [rika](https://github.com/b-zago/rikami) CLI

## Labels

Labels for prod/staging are handled entirely by the [rikami-charts](https://github.com/b-zago/rikami-charts) library automatically when the right values are provided.

I've chosen the following labels for all of my prod/staging workloads:

- **runs-on** - what is running on a resource (e.g. nginx, python, nodejs for deployments/pods). For resources like secrets that can be attached to many things, the `runs-on-*` form applies instead (e.g. `runs-on-webserver=true`); either way you pass `runsOn` in Helm.
- **part-of** - the higher level application this resource belongs to (e.g. portfolio)
- **instance** - unique name identifying a resource within an application
- **component** - the component within the architecture, kept fairly broad (e.g. server, db, cache, httproute, service)

For argocd applications and appsets:

- **part-of** - which app watches this one in the app-of-apps tree
- **appset** - if this application was created via an appset, which appset it belongs to
